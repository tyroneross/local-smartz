## Best path

### Shipping order (Reflection → Orchestrator-Workers → Parallelization)

Inverts the user's mental order. Rationale below.

#### A. Reflection (~3 days)
- `src/localsmartz/patterns/reflection.py`
- Single agent, Ω≤3 self-critique iterations. Agent generates → agent reflects on own output with structured rubric → revises.
- Same-model-two-prompts (like critic_loop on mini), no co-resident model requirement. Works on 24GB floor.
- Persistence: append reflections to `threads.append_entry` with `kind="reflection"`. Makes them queryable in later threads.
- Span tree: `ls.pattern.run[pattern=reflection]` → `ls.reflection.<n>` children with `ls.reflection.{index, rubric_score, revision_reason}`.
- Ship first because: cheapest, unlocks mini-tier quality, validates the span-shape regression test pattern at scale, exercises `threads.append_entry` under the reflective-memory workload before orchestrator-workers adds artifact write pressure.

#### B. Orchestrator-Workers (~1 week)
- `src/localsmartz/patterns/orchestrator.py`
- **One level only** — hardcode the constraint. No nested subagents. F7 guard.
- Lead agent decomposes into subtasks + writes plan to `todo.md` (Manus pattern). Workers run in sequence on mini, parallel on standard+. Each worker writes output via `artifacts.register`. Orchestrator reads **artifact IDs + summaries only**, never raw worker output. This is the research-doc-specified F7 mitigation.
- **Two implementation paths**:
  - **Path A (cloud + standard tier)**: use DeepAgents `sub_agents` param + `task` tool. Native, ~20 lines like Anthropic. Honors the RunnableRetry guard because the runner is a LangChain chat model built inside our `_create_*_model` functions.
  - **Path B (mini tier + provider-agnostic)**: iterate workers via `AgentRunner.run_turn` directly, skipping DeepAgents. Ensures mini path doesn't load full DeepAgents machinery.
  - Orchestrator protocol choice (Path A vs B) happens at pattern-runtime, not pattern-definition. Config key: `profile.use_deepagents_subagents` defaulting True on standard+/cloud, False on mini.
- Tier gate: standard+ required by default; mini tier shows warning + requires explicit override.
- Artifacts surface: register each worker output; `GET /api/artifacts?thread_id=...` already works (threads + artifacts modules are wired).

#### C. Parallelization / Ensemble (~3 days)
- `src/localsmartz/patterns/parallel.py`
- N samples of same prompt → synthesizer agent picks best or merges.
- On mini: sequential (same model, different temperatures or personas) — not real parallelism, but still useful for ensemble voting. Use `asyncio.as_completed` with a single-slot semaphore.
- On standard+: true concurrent via `asyncio.gather` with a semaphore bounded by tier-appropriate concurrency (2 for standard, 3 for full).
- Synthesizer: can be same-model-larger-prompt (cheap) or stronger-model-different-ref (quality). User-selectable.
- Span tree: `ls.pattern.run[pattern=parallel]` → `ls.sample.<n>` children + `ls.synthesize` child.

## Why

Ranked by "why this ordering" and "why these choices":

1. **Reflection first because** it's the highest-ROI low-risk add. It unblocks mini-tier quality without requiring multi-model coordination. It reuses `threads.append_entry` — validates the reflective-memory write volume before orchestrator-workers starts writing artifacts. Ω≤3 bound keeps compound error bounded. Same-model-two-prompts shape means zero new model-coordination risk.
2. **Orchestrator-Workers second because** it's the load-bearing Type III pattern (research doc). The Anthropic catalog shows 90.2% uplift when done right, and we have the filesystem substrate (`artifacts.py`) already. One-level-only is a hard constraint that prevents the worst compound-error case; we enforce it in code (not a recommendation). The dual-path (DeepAgents vs runner-only) honors the RunnableRetry guard AND gives us mini-tier feasibility.
3. **Parallelization last because** it has the most variable cost (concurrent model loads on mini tier = thrash; F2-like but worse). Shipping after Orchestrator-Workers means the pattern-tier gating UX is already polished from that build. Ensemble on mini will be "sequential samples" — useful but not the real concurrency story; defer the real parallel mode until standard+ is a common target.
4. **Reuse `artifacts.py` because** rebuilding it would duplicate the file-lock + id-generation + per-thread scoping. It's 145 LOC of tested primitive.
5. **One-level-only in code, not as a recommendation, because** Anthropic's own lab notes flag cascading hallucination as the killer of nested subagents. Leaving it as "please don't do this" invites someone (possibly future Claude) to relax it when a build seems to need it.
6. **Logit masking adapter is optional scope because** agent-builder catalog flags it as high-leverage but it requires moving beyond Ollama's API surface. Start with prompt-level constraint; upgrade only if mini-tier eval shows F4 recurring.

## Risks

From the F1–F24 list + new Phase-3-specific:

**Inherited risks being stressed by Phase 3**
- **F1 OOM on mini** — parallelization's concurrent model loads make this worse. Mitigate: hard-cap concurrent runner calls via semaphore, tier-gated default = 1 on mini.
- **F2 model-swap thrash** — orchestrator + workers with different model refs = thrash on mini. Mitigate: same resolve.py rule as critic_loop; force workers to share orchestrator's model on mini, log INFO.
- **F4 tool hallucination** — more prominent with more subagent calls. Mitigate: preserve per-role tool allow-list; add logit masking adapter.
- **F5 compound error** — hits Reflection hardest (Ω iterations multiply). Mitigate: Ω hard cap at 3, budget.max_turns enforced, early-exit on rubric score threshold.
- **F7 orchestrator telephone-game** — the explicit target pattern for Phase 3. Mitigate: filesystem context isolation (artifacts.py), orchestrator reads summaries only, one-level constraint in code.
- **F10 pattern divergence local vs cloud** — new patterns compound this. Mitigate: golden-task suite already runs on both providers; regression gate when adding Phase 3 patterns to the suite.

**New Phase-3-specific risks**
- **P1 Nested subagent cascade** — a subagent tries to spawn its own subagent. Mitigate: hardcode depth=1 check in both DeepAgents path and runner-only path; raise if violated.
- **P2 Artifact sprawl** — workers register many artifacts per run, `.localsmartz/artifacts/` grows unbounded. Mitigate: TTL + prune in artifacts.py (already has manifest format, just add prune logic); surface disk usage via existing doctor.
- **P3 Reflection stall** — agent enters a "revise forever, never converge" pattern. Mitigate: rubric score must improve by ≥ delta between iterations OR loop aborts with `verdict=stagnant`. Already have LoopDetector we can extend.
- **P4 Parallel samples identical** — low-temperature sampling produces near-duplicate replies, defeating ensemble value. Mitigate: require temperature ≥ 0.3 for parallel pattern, OR persona-style prompts that vary the framing.
- **P5 OTel span explosion** — orchestrator + 5 workers + 3 reflections in one run = 20+ spans. Phoenix handles it fine at small scale (verified: 14ms live trace today); at scale SQLite store grows linearly. Mitigate: document retention + prune recipe; consider sampling at 1.0 for dev, 0.1 for long runs.
- **P6 `create_deep_agent` cache poisoning by subagent variation** — every new SubAgent definition variation re-creates the cache key. Mitigate: define subagent structs at module level, not per-run.

**Unknowns carried from Phase 2 still open**
- U2 `qwen3.5:9b` tag existence — Phase 3 adds more model references; catalog refresh valuable before shipping. Today's ollama pull on this machine: `qwen3.5:9b` not attempted as default; `gemma4:26b` present; `gpt-oss:20b` present but **not in catalog** — catalog gap.
- U11 Groq vs OpenAI dialect differences — parallelization across providers will expose this immediately if present.
